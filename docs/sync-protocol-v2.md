# SecretBase Sync Protocol V2

本文档定义 SecretBase V5.3 的全量快照同步协议。V2 面向只稳定支持
`HTTPS`、Basic Auth、`MKCOL`、`PUT`、`GET`、`PROPFIND` 和 `DELETE` 的普通
WebDAV 服务，包括坚果云。V2 与 V1 完全隔离，V1 的强 ETag 和可变 head 语义
不被修改。

## 1. 设计目标

- 远端只保存端到端加密的完整 Vault 快照，不上传字段名或字段值明文。
- 不依赖 ETag、`If-Match`、`If-None-Match` 或可变共享索引。
- 两台设备并发写入时使用不同的不可变对象，不互相覆盖。
- 同步前发现远端所有可见提交，验证提交关系，再自动合并或交给用户处理冲突。
- 网络中断、进程终止或重复重试不会破坏已经成功写入的快照。
- V1 用户继续使用 V1；V2 新配置和显式迁移使用独立远端目录。

## 2. 远端布局

```text
secretbase-sync-v2/
  <vault_id>/
    <space_id>/
      snapshots/
        <device_id>/
          <generation>-<snapshot_id>.sbs
```

`vault_id`、`space_id`、`device_id` 和 `snapshot_id` 都是规范化小写 UUID。
文件名中的 generation 只用于发现和诊断，最终以解密后的提交内容为准。
每个提交路径只写入一次，客户端使用随机 UUID 生成路径，不向已有路径发送覆盖
操作。远端服务可以看到 UUID、对象数量、大小和访问时间，但看不到密码库内容。

V2 没有 `head.sbh`。目录中的提交构成一个有向无环图（DAG）：每个提交包含
零个或多个 parent，当前 frontier 是没有子提交的提交集合。

## 3. 加密对象

V2 bundle 使用 `SBS2` magic、版本 `0x02`、对象类型 `0x01`、随机 12 字节
AES-GCM nonce，以及 `ciphertext || 16-byte authentication tag`。明文先使用
Gzip 压缩，单个密文和解压后的 JSON 均限制为 64 MiB。

一次远端发现最多接受 1000 个快照；累计下载密文和累计解密 JSON 分别限制为
256 MiB，frontier 最多 32 个。任一限制超出时停止同步，并要求先在已连接设备上
处理冲突或压缩历史，不能截断历史后继续合并。

AAD 为：

```text
UTF8("SecretBase Sync V2") || 0x00 ||
UTF8(vault_id) || 0x00 || UTF8(space_id) || 0x00 || UTF8(snapshot_id)
```

快照明文结构：

```json
{
  "schema_version": 2,
  "protocol": "snapshot-dag",
  "vault_id": "uuid",
  "space_id": "uuid",
  "snapshot_id": "uuid",
  "generation": 7,
  "parents": ["uuid"],
  "created_at": "UTC ISO-8601",
  "device_id": "uuid",
  "device_name": "设备名称",
  "document": { "完整 Vault V1 文档": true }
}
```

### 提交规则

- 初始提交 generation 为 1，parents 为空。
- 后续提交 generation 为所有 parent generation 的最大值加 1。
- parents 去重，最多 32 个；合并提交可以有多个 parent。
- snapshot_id 每次随机生成，客户端不能复用路径。
- 上传后必须重新 GET 并逐字节校验本次密文；校验失败不得把提交记为成功。
- 任何无法解密、身份不一致、parent 缺失、循环、generation 回退或文件名不符
  的远端对象都会使本次同步停止，不得用空密码库初始化或静默丢弃历史。

## 4. 发现与合并

1. `PROPFIND Depth: 1` 读取 snapshots 目录，得到设备目录；带 XML 命名空间前缀的
   `DAV:` 响应按本地元素名称解析。
2. 对每个设备目录再次 `PROPFIND Depth: 1`，得到快照文件。
3. 下载并验证新增或未知快照，构造 DAG，计算 frontier。
4. 根据本机加密基线中的 frontier 判断远端是否前进。
5. 只有本机修改时，创建以本机基线为 parent 的新提交。
6. 只有远端修改时，直接应用远端 frontier 文档。
7. 两端都修改时，以共同基线执行现有三方合并；无冲突则创建多 parent 合并提交，
   有冲突则只返回脱敏摘要并等待用户选择。

远端存在多个 frontier 时，客户端先找共同祖先，再逐分支合并。若某一分支发生冲突，
客户端只提交当前一批选择，剩余分支保留在一次性冲突上下文中；所有分支完成前不得
创建最终合并提交。冲突摘要只包含
条目标题、更新时间、字段数量和变化区块，不包含字段名或字段值。冲突计划绑定
解锁会话、Vault revision、space_id 和发现时的 frontier；任何本机或远端变化都会
使旧计划失效。

## 5. 失败恢复与历史

- 上传快照后应用崩溃只会留下孤儿对象；下一次发现时仍可验证并纳入 DAG。
- 加入同步空间先事务性提交本机配置、远端文档和基线，再按需发布本机合并快照。
  如果首次发布因网络失败，本机仍保持已加入状态，清除未完成上传令牌，并在下一次
  同步继续发布；界面不得回到加入表单或把半完成计划显示为可用配置。
- 不自动删除历史或孤儿快照。没有共享 head，客户端无法知道另一台离线设备是否
  仍以旧提交作为基线。
- 提供显式的“压缩并清理”操作：客户端先确认本机文档、远端 frontier 和远端折叠文档
  与本机基线完全一致，然后在新的 `space_id` 下创建一个 generation=1 的根快照，
  原子切换本机配置，再逐对象清理旧空间。新空间与旧空间隔离，避免无 ETag 服务上
  直接删除正在被离线设备引用的对象。
- 压缩后其他设备不会自动发现新空间，必须使用新恢复码重新加入。新空间创建成功后，
  旧空间清理失败只报告“待清理”，不回滚或删除新空间；删除前后均重新枚举，遇到
  新增对象或删除失败立即停止并显示诊断。
- 恢复历史版本不是覆盖旧对象，而是以所选文档创建一个新的提交。
- 远端删除是逐对象的显式危险操作；清理前先验证完整 DAG，清理中保护未知对象，
  删除已知对象后再次枚举，发现并发新增或非协议对象时停止。本机 Vault 不随之删除。
- 基础 WebDAV 没有可用于整个目录树的原子 CAS。即使客户端在删除前后重新枚举，
  最后一次“目录为空”检查与删除目录之间仍存在服务端无法消除的竞态。因此 V2 不保证
  远端删除事务原子性，只保证不会因清理失败删除本机 Vault，也不会把检测到的变化
  误报为成功；需要严格事务删除的部署应使用支持更强服务端语义的存储。

## 6. 配置与恢复码

V2 配置包含 `protocol_version: 2`、`vault_id`、`space_id`、独立随机 256 位
sync key、WebDAV 凭据和 device identity。配置与本机基线仍使用 Vault 用途隔离
密钥加密保存。

恢复码格式为 `SBSYNC2-...`，包含版本、vault_id、space_id、sync key 和校验和。
配对 URI 包含协议版本、Vault/space 身份、V2 恢复码、WebDAV 地址和用户名，不包含
WebDAV 密码。V2 恢复码本身已经包含同步空间身份、同步密钥和校验和，因此新 URI 不再
重复携带裸 `key` 参数；客户端仍可兼容 V5.3.0 生成的旧 `key` 形式。恢复码和 URI
等同于同步密钥，只能在再次验证主密码后显示，不能写入日志、诊断或浏览器持久化存储。

## 7. 服务兼容性与重试

V2 只要求基础 WebDAV 方法，不发送条件写入头。传输层对超时、429 和临时 5xx 做
有限次数指数退避，尊重不超过 5 秒的 `Retry-After`；认证失败、TLS 错误、重定向、
4xx 参数错误不重试。生产环境仍强制 HTTPS，不提供忽略证书错误的开关。

自动同步在解锁及非同步 Vault revision 变化后使用 5 秒防抖，手动与自动同步必须进入
同一全局互斥区。Vault 锁定时客户端取消活动 WebDAV 请求并清除待处理敏感上下文；
恢复、轮换、压缩和远端删除均要求再次验证主密码，压缩、轮换和删除使用一次性确认
令牌防止重复提交。

V2 测试必须覆盖无 ETag、拒绝条件请求、PROPFIND 分页/乱序、429/503、延迟、上传
中断、重复同步、双设备和三设备并发，以及远端对象损坏和删除竞态。
