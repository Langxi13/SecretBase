# SecretBase App 化长期规划

本文档记录 SecretBase 从自托管 Web 工具逐步演进为桌面应用和手机应用的长期方向。它不是具体开发任务清单，而是用于后续版本需求讨论、边界确认和实施计划拆分的产品路线基准。

## 1. 长期目标

SecretBase 的长期目标是成为一个以个人本地数据安全为核心的离线密码库：

- 可以继续作为自托管 Web 服务运行。
- 可以打包为 Windows 桌面应用，优先满足完全本地离线使用。
- 未来可以打包为手机 App，使用同一套 vault 数据规则和加密规则。
- 数据迁移优先依赖手动加密备份文件，而不是云同步或账号系统。

当前阶段不急于实现打包功能。更重要的是先把运行模式、数据目录、安全边界、迁移方式和版本边界讨论清楚。

## 2. 当前基线

当前 SecretBase 的基础形态如下：

- 后端使用 FastAPI，负责认证、vault 读写、条目管理、标签、回收站、备份、导入导出、工具报告和 AI 解析。
- 前端使用 Vue 3 CDN、原生 JavaScript 和原生 CSS，无 npm 构建链。
- 数据以本地加密 vault 文件保存，不依赖数据库。
- 已具备手动备份、自动备份、备份中心、导入导出和三步恢复流程。
- 当前部署模型仍以自托管 Web 为主，推荐在 nginx、HTTPS 和额外访问控制之后运行。

这些能力会作为 App 化的基础复用。短期内不改变 vault 文件格式，不重写核心密码库能力。

## 3. 总体路线

App 化路线采用渐进式演进：

1. 先支持桌面本地运行模式。
2. 再完成 Windows 桌面离线包。
3. 然后补齐桌面产品化体验。
4. 最后再为手机 App 抽象共享 vault 核心。

桌面首版选择 `PyInstaller + pywebview`，原因是它能最大程度复用现有 Python 后端和静态前端，首版改造面较小。手机端不计划内嵌 Python/FastAPI，长期应通过共享 vault 核心实现离线能力。

## 4. 版本阶段

### V3.0 桌面基础层

V3.0 的目标是让当前系统具备桌面运行的基础条件，但还不追求正式安装包体验。

重点包括：

- 增加 `server` 和 `desktop` 两种运行模式。
- 桌面模式使用用户目录保存数据、备份、日志和设置。
- Windows 默认数据根目录为 `%LOCALAPPDATA%\SecretBase\`。
- 桌面模式后端只监听 `127.0.0.1`，端口由启动器动态分配。
- 前端 API 地址由启动器注入，不再依赖固定 `127.0.0.1:10004`。
- 保持服务器部署模式的默认行为不变。

V3.0 的成功标准是：在开发环境中可以用桌面模式启动本地后端和前端，并且不读写仓库内的真实运行数据目录。

V3.0 的详细规划见 [SecretBase V3.0 桌面基础层规划](v3-desktop-foundation.md)。

### V3.1 Windows 桌面 MVP

V3.1 的目标是交付第一个可用的 Windows 离线桌面版本。

重点包括：

- 使用 PyInstaller 打包 Python 后端、静态前端和 pywebview 桌面壳。
- 启动桌面应用时自动拉起本地后端。
- 桌面窗口打开后可以完成初始化、解锁、条目 CRUD、搜索、标签、回收站、备份中心、导入导出和恢复。
- 关闭桌面窗口时停止本地后端进程。
- 应用包内不得包含真实 vault、日志、`.env`、备份文件或用户数据。

V3.1 不包含自动更新、代码签名、系统托盘、开机启动、浏览器自动填充和云同步。

V3.1 的详细规划见 [SecretBase V3.1 Windows 桌面 MVP 规划](v3.1-windows-desktop-mvp.md)。

### V3.2 桌面产品化增强

V3.2 的目标是让桌面应用更接近日常可长期使用的状态。

可考虑的能力包括：

- Windows 安装器。
- 应用图标和基础版本信息。
- 启动失败提示。
- 数据目录打开入口。
- 桌面模式健康检查，包括后端进程状态、数据目录权限、端口冲突、备份目录可写性。
- 可选的系统托盘和退出前锁定。

本阶段仍不记住主密码，不把主密码保存到系统钥匙串。

### V4.0 共享 vault 核心准备

V4.0 的目标是为手机 App 和未来跨平台能力建立稳定的数据核心边界。

重点包括：

- 固化 vault 文件格式说明。
- 建立跨语言兼容性测试样例。
- 明确加密、解密、备份读取、备份写入、恢复校验的输入输出规则。
- 评估是否将 vault/crypto/storage 核心抽象为 Rust 或其他可跨桌面和手机复用的实现。

本阶段不急于替换现有 Python 后端，而是先确保现有格式和行为可以被另一个实现准确复现。

### V5.0 手机 App

V5.0 的目标是实现完全离线的手机 App。

初版手机 App 只关注核心密码库能力：

- 本地创建和解锁 vault。
- 条目浏览、搜索、查看、复制和编辑。
- 标签和回收站基础操作。
- 手动加密备份导入和导出。
- 与桌面版通过加密备份文件进行迁移。

手机 App 初版不做账号系统、云同步、多设备实时同步、浏览器自动填充和团队协作。

## 5. 数据和迁移原则

SecretBase 的数据边界继续以加密 vault 和加密备份为核心：

- vault 文件格式保持稳定，不能为了 App 化随意破坏兼容性。
- 手动加密备份是桌面和手机之间的第一迁移媒介。
- 明文 JSON 只作为显式确认后的导出能力，不作为推荐迁移方式。
- 自动备份只负责本机恢复安全，不承担跨设备同步职责。
- 任何格式变更都必须先有备份、恢复和兼容性测试。

## 6. 安全边界

App 化不能降低当前安全模型：

- 主密码不保存明文。
- 不提供“忘记主密码后找回”能力。
- 桌面首版不记住主密码。
- 桌面后端只监听本机地址。
- 桌面模式默认不暴露局域网访问。
- 应用包和安装器不得包含用户数据。
- 日志继续脱敏，避免记录密码、token、API key 和备份主密码。

桌面应用的便利性必须服从本地数据安全边界。

## 7. 明确非目标

以下能力不进入近期 App 化路线：

- 多用户、组织、团队协作和共享 vault。
- 云同步、官方云服务或账号体系。
- 浏览器插件和自动填充。
- 自动更新和发布签名作为桌面 MVP 的必选项。
- 手机端内嵌 Python/FastAPI。
- 使用明文 JSON 作为主要跨设备迁移方式。
- 保存或自动填充主密码。

## 8. 下一步讨论重点

下一轮应只详细规划 V3.0 桌面基础层，不急于进入打包实现。

需要讨论并确认：

- `server` 和 `desktop` 运行模式如何共存。
- 桌面数据目录的具体结构。
- 桌面启动器如何选择端口、注入前端 API 地址、关闭后端。
- 桌面模式是否仍读取 `.env`，以及哪些配置应被桌面启动器覆盖。
- 备份、日志、设置文件在桌面模式下的路径规则。
- V3.0 必须补齐哪些测试，才能进入实现阶段。

这些问题的第一版讨论结果已整理到 [SecretBase V3.0 桌面基础层规划](v3-desktop-foundation.md)，后续可以继续在该文档基础上细化实施计划。
