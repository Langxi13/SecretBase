# SecretBase 前端视觉大换代设计规格

日期：2026-07-07

## 1. 目标

本轮改造把 SecretBase 的前端从当前“可用的深色卡片工作台”提升为更精致、更稳、更适合长期使用的中文密码库界面。主方向采用“高级暗色型”，吸收“克制专业型”的层级和留白：界面要更好看，但不能变成装饰优先的展示页。

本轮只改前端 UI、样式、可见交互细节和暴露出的前端问题。后端接口、vault 数据结构、认证模型、备份语义、AI 配置语义不变。

## 2. 范围

包含：

- 重做主题变量、深色视觉质感、按钮、输入框、标签、卡片、状态提示和空状态。
- 优化桌面端主工作台：侧边栏、顶部栏、搜索筛选区、条目卡片、底部批量操作栏。
- 优化移动端体验：搜索筛选可读性、卡片宽度、底部操作区、模态框高度、按钮换行和触达。
- 优化主要弹窗：条目详情、创建/编辑、AI 智能录入、备份中心、设置、工具、标签管理、回收站、导入预览和恢复向导。
- 所有可见新增或调整文案使用中文。
- 在视觉 QA 中修复前端隐藏问题，重点包括溢出、遮挡、对比不足、长文本断行、移动端不可触达、加载态缺失、状态提示误导和点击外部关闭失效等前端问题。

不包含：

- 不引入 React、Vue SFC、Vite 或 npm 构建链。
- 不重写为三栏详情工作台。
- 不新增后端接口，不改变 API 路径和请求/响应语义。
- 不改变搜索范围、导入导出、备份恢复、AI 解析、自动锁定等已验收行为。
- 不加入英文界面或多语言系统。

## 3. 视觉方向

默认视觉以深色为主，浅色主题保持可用但不是本轮主秀场。深色主题避免单一蓝紫或纯黑压迫感，使用深墨色底、冷蓝主色、青绿色状态色和少量暖色警示。卡片边界要清楚，阴影和高光要克制，不能靠大面积渐变或装饰光斑撑质感。

关键原则：

- 密码库是高频工具，不做营销页、英雄区或浮夸背景。
- 信息密度保持偏高，但层级更清楚：主操作、筛选、状态、列表、危险操作各自有明确视觉权重。
- 所有卡片、按钮、输入框、弹窗遵循稳定尺寸和响应式约束，避免 hover、加载文本或长文件名导致布局跳动。
- 图标保留当前无构建链约束下的字符图标或内联图标，不引入图标依赖。

## 4. 界面结构

桌面端继续使用现有“左侧导航 + 右侧工作区”结构。左侧导航要更像应用侧栏：品牌区、主操作、视图区、标签区、管理区分层清楚，当前状态突出但不刺眼。右侧工作区包含顶部工具栏、搜索筛选面板、列表状态提示和条目网格。

移动端继续使用顶部栏、搜索筛选区、单列条目卡片和底部操作区。移动端不强行压缩桌面侧栏，重点保证筛选控件可读、按钮不溢出、底部栏不遮挡主要内容。

条目卡片继续展示标题、网址提示、标签、字段摘要、星标、选择框和复制/打开/编辑/删除操作。视觉上要让标题、标签、敏感字段、普通字段和危险动作一眼区分。可复制字段默认仍显示掩码，不改变安全行为。

## 5. 组件改造

主题变量：扩展现有 CSS 变量，补齐表面层级、微边框、焦点环、危险/警告/成功的低饱和背景、固定尺寸、透明表面和文本层级。

按钮：统一主按钮、次按钮、危险按钮、小按钮、图标按钮和底部操作按钮的尺寸、焦点态、禁用态和加载态。危险操作使用低饱和红色，不和普通删除文本混淆。

输入控件：搜索框、日期、下拉、文本域、标签输入和文件选择周边容器都继承主题；暗色下保持 `color-scheme: dark`，避免原生控件文字不可读。

筛选区：搜索范围 chip、排序控件和高级筛选统一成分组面板。当前筛选 chip 要明显可见，并提供单项移除和一键清除状态。

模态框：桌面端加宽复杂弹窗，移动端使用接近全屏的底部友好布局；弹窗头尾固定时要保证正文可滚动，不能遮挡确认按钮。长文件名、长标签和长错误消息必须换行。

反馈组件：空状态、加载态、Toast、警告面板和信息面板统一视觉语言。AI 失败、备份恢复、导入冲突等高风险反馈要更醒目，但不改变原有流程。

## 6. 数据流和状态边界

前端继续复用 `frontend/js/api.js`、`frontend/js/store.js` 和 `frontend/js/app.js` 的现有状态流。视觉改造不得改变以下边界：

- Token 仍通过 `X-SecretBase-Token` 发送，前端使用 `sessionStorage`。
- 条目列表仍通过现有筛选参数加载，搜索词必须配合已选搜索范围。
- 条目详情仍按需获取明文字段，可复制字段列表默认掩码。
- 备份中心仍区分手动、自动和旧备份，恢复仍走三步确认。
- AI 解析仍依赖设置中的 Base URL、API Key 和模型，失败后保留原文并可转手动录入。

可以做的结构性整理仅限前端可维护性：在不引入构建链的前提下，允许提取少量纯函数、常量或局部渲染辅助，前提是测试和手动验收能覆盖行为不变。

## 7. 错误处理和隐藏问题修复

本轮把“视觉 QA”作为发现隐藏前端 bug 的手段。发现的问题优先按以下规则处理：

- 如果是前端样式或状态同步问题，直接在本轮修复。
- 如果需要后端接口变化，记录为后续问题，不在本轮扩展。
- 如果修复会改变已验收业务行为，先保留行为，只改提示或布局。

重点检查：

- 未解锁、自动锁定、401 返回时的页面切换和提示是否一致。
- 标签浏览、复制菜单、筛选面板等浮层点击外部是否能关闭。
- 备份中心长文件名、导入预览长标题、AI 长文本、标签长名称是否溢出。
- 移动端底部栏是否遮挡内容，弹窗按钮是否可触达。
- 浅色/暗色/跟随系统下输入框、日期控件、下拉框是否可读。
- 加载中、禁用中和错误状态是否有明确中文反馈。

## 8. 验收方式

自动验证：

- `node --check frontend/js/api.js`
- `node --check frontend/js/store.js`
- `node --check frontend/js/app.js`
- 如果后端相关文件未改，后端编译不是本轮必须项；如果改到后端，再运行 `python -m compileall backend`。

手动/截图验收：

- 启动本地前端和后端或桌面模式，检查未初始化、锁定、主界面、详情、新建/编辑、AI、备份、设置、工具、标签、回收站、导入预览和恢复向导。
- 至少检查桌面宽屏、平板窄宽和手机宽度。
- 检查暗色、浅色和系统主题。
- 用长标题、长标签、长字段名、长备份文件名和多条 AI 结果验证不溢出、不遮挡、不跳动。

## 9. 成功标准

- 用户第一眼能感到界面明显更精致、更像成熟应用，而不是只换了颜色。
- 所有已有核心入口仍可找到，中文文案一致。
- 桌面端信息层级更清楚，移动端不再显得拥挤或难触达。
- 暗色主题质感提升，浅色主题保持无明显破损。
- 视觉 QA 暴露的前端问题已修复或明确记录为后续项。
